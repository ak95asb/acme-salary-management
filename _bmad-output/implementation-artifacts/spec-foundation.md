## Intent

**Problem:** The ACME salary management project has no code yet â€” only planning artifacts. Nothing can be built until the monorepo scaffold, database schema, shared type contracts, and Express skeleton are in place.

**Approach:** Scaffold a Turborepo pnpm monorepo at the project root with `apps/api` (Express 5 + TypeScript) and `packages/types` (shared Zod schemas). Define the full Prisma schema (7 models), run the initial migration, create a passing health endpoint, and wire Vitest. This is the foundation every subsequent goal builds on.

## Boundaries & Constraints

**Always:**
- pnpm workspaces + Turborepo; package manager is pnpm only (no npm/yarn lockfiles)
- TypeScript strict mode in all packages (`"strict": true`)
- Single Prisma client exported from `apps/api/src/lib/prisma.ts`
- All env vars validated at startup via `@t3-oss/env-core` + Zod in `apps/api/src/env.ts`
- Prisma schema in `apps/api/prisma/schema.prisma`; SQLite provider for dev
- Monetary fields: `Decimal @db.Decimal(15,4)` â€” never `Float`
- `.env` gitignored; `.env.example` committed with all required key names (values empty)
- Git commit at end: `chore: scaffold monorepo foundation with Prisma schema and Express skeleton`

**Ask First:**
- If pnpm or Turborepo CLI is unavailable in the environment

**Never:**
- `npm install` or `yarn` â€” pnpm only
- `new PrismaClient()` outside `lib/prisma.ts`
- Next.js frontend scaffold (Goal 7)
- Auth or business logic (Goals 2â€“6)

## Design Notes

**Decimal as string in Zod schemas:** Salary `amount` is `z.string().regex(/^\d+(\.\d{1,4})?$/)` in `createSalarySchema` â€” not `z.number()`. This is because Prisma's `Decimal` type roundtrips cleanly with string serialisation. The service layer converts to `new Decimal(amount)` before writing to DB.

**Env validation order:** `src/env.ts` must be the first import in `src/index.ts`. If it throws, the process exits before the Express app or Prisma client are constructed â€” no partial startup.
